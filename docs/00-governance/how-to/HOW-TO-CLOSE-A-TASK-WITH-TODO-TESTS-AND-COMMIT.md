# HOW TO CLOSE A TASK WITH TODO, TESTS, AND COMMIT

## 1. Purpose

本文档说明：当代码改动已经完成后，如何判断当前任务是否可以进入收口。  
目标是让开发者和 AI 用同一套最小闭环完成 `TODO.md`、测试、文档和提交的收尾，而不是停留在“代码看起来差不多完成了”。

## 2. Scope

当前范围：

- 代码改动已经完成后的收口判断
- `TODO.md` 项是否删除、拆分或收窄
- 最小测试检查清单
- 最小文档同步检查清单
- 小步提交与 `commit message` 收口

不在范围内：

- 不指导具体业务功能如何实现
- 不替代测试写法教程
- 不替代 Git 基础使用教程
- 不处理需求尚未固定时的任务启动问题

## 3. Bounded Context

本仓库把“任务完成”定义为一个闭环，而不是单纯代码已改完。

- `TODO.md` 是执行队列，不是完成历史
- 完成历史保留在 Git commit / PR 中
- 代码、测试、文档、`TODO.md` 清理和提交应尽量收敛成同一条可解释的变更链路

因此，收口阶段的重点不是继续扩写功能，而是判断当前改动是否已经形成一个可验证、可回放、可提交的最小闭环。

## 4. When To Use

- 你已经完成当前任务的主要代码改动
- 你准备删除、拆分或收窄 `TODO.md` 中的对应任务项
- 你准备判断是否还需要补测试或同步文档
- 你准备整理 commit 边界和 `commit message`

## 5. Do Not Use For

- 功能尚未进入可运行状态
- 需求、边界或主流程仍在讨论中
- 当前改动仍包含多个未拆开的不相关判断
- 你只是想快速提交临时草稿，不准备收口任务

## 6. Pre-Checks

开始收口前，固定先判断以下问题：

1. 当前任务的实现目标是否已经达到可运行、可检查状态。
2. 当前改动是否仍存在未完成的剩余范围。
3. 当前改动是否影响测试、文档或任务入口。
4. 当前工作区中是否混入了其他不相关修改。
5. 当前提交是否能够用一句明确的话解释“这次到底改了什么能力”。

只要以上任一问题答案不明确，就不要进入收口。

## 7. Steps

### 7.1 Confirm Task Closure Status

先判断 `TODO.md` 对应任务项是否真的完成。

- 任务目标已经完成，且没有剩余未实现范围：删除对应 `TODO.md` 项
- 任务只完成一部分，但剩余范围仍清楚存在：不要删除；将任务拆分或收窄为剩余未完成内容
- 当前只是局部验证通过，但需求或边界尚未稳定：不要删除；保持原任务项不动

判断原则：

- 不得因为“代码已经写了一部分”就删除整项
- 不得把“待讨论”状态误当成“已完成”
- 不得把一次大任务的剩余部分藏到 commit 历史之外

### 7.2 Run The Minimum Test Check

收口前至少检查当前改动影响了哪些测试层次。

最小检查清单：

- Controller 是否新增或改变了公开 HTTP 签名、校验、权限、页面跳转或异常响应
- Service 是否改变了业务流程、事务边界、状态流转、跨 DAO 编排或结果模型
- DAO / Mapper 是否改变了查询、分页、SQL、实体映射或持久化约束
- static 是否改变了 API 文档、上传访问或其他运行支撑资源
- 是否改动了架构规则、命名规则或其他可能影响静态检查的内容

判断结果：

- 行为变化已发生：同步补或更新对应测试
- 只有纯机械改名、注释、格式整理，且不影响行为和规则：可不新增测试，但必须确认没有误触行为边界

### 7.3 Run The Minimum Document Sync Check

收口前至少检查当前改动是否要求同步文档。

最小检查清单：

- 需求口径变化：检查 `10-requirements/*-REQUIREMENTS.md`
- 治理规则或固定做法变化：检查 `docs/00-governance/*.md`
- 数据库结构、字段、查询语义变化：检查 `20-database/*-DATABASE-DESIGN.md`
- AI 默认读取入口或任务路由变化：检查 `docs/AGENT.md`

判断原则：

- 改动影响文档定义的固定口径时，必须同步文档
- 仅代码实现补齐、但不改变既有文档口径时，不为“看起来完整”而顺手改无关文档

### 7.4 Decide Commit Boundaries

commit 粒度使用混合判断：

- 先保证单个 commit 语义完整
- 再避免把不相关判断混在同一个 commit 中

可以放在同一个 commit 的内容：

- 同一能力的代码、测试、文档同步
- 对应 `TODO.md` 项删除
- 为同一判断服务的命名或装配修正

应该拆成多个 commit 的内容：

- 不同业务模块的独立能力改动
- 同一任务中彼此无直接因果关系的两类判断
- 纯治理补丁与业务功能改动，如果它们不是同一个闭环

### 7.5 Write The Commit Message

`commit message` 必须同时表达：

- 改动类型
- 业务域或治理域
- 具体能力变化

本仓库固定格式：

`Type(domain): 中文说明`

检查清单：

- `Type` 是否准确表达这是 `Feat`、`Fix`、`Docs`、`Test`、`Refactor` 等哪类变化
- `domain` 是否能稳定定位业务域、模块或治理域
- 中文说明是否点出具体能力，而不是只写“调整”“修改”“优化”

## 8. Files To Touch

本任务通常新增或修改以下文件类型：

- 根目录 `TODO.md`
- 当前改动已经影响的测试文件
- 当前改动已经影响的需求/治理/数据库文档
- 本次提交涉及的业务代码文件

本任务通常不会新增新的业务结构层，也不应为了收口动作额外扩写无关功能文件。

## 9. Common Mistakes

- 任务只完成一部分，就删除整个 `TODO.md` 项
- 代码行为变了，但因为“先能跑”而不补测试
- 规则口径已经变化，却没有同步需求文档或治理文档
- 多个不相关判断混在同一个 commit 中
- `commit message` 只写“调整”“修改”“优化”，看不出具体能力变化
- 为了让提交显得“大而完整”，顺手夹带无关清理

## 10. Verification

收口前固定自检：

1. `TODO.md` 是否准确反映剩余未完成内容。
2. 本次改动影响到的最小测试层次是否已经检查。
3. 本次改动影响到的最小文档类型是否已经检查。
4. 当前工作区是否只包含准备提交的同一闭环修改。
5. 当前 commit 是否能被一句明确的话解释。

只要其中任一项不能明确回答“是”，当前任务就还不应结束。

## 11. Commit Guidance

优先使用一个能独立说明闭环的 commit 完成收口。

推荐顺序：

1. 完成代码改动
2. 补齐或更新必要测试
3. 同步必要文档
4. 删除、拆分或收窄 `TODO.md` 项
5. 提交带语义的 commit message

如果发现当前改动包含多个不相关判断，应先拆开再提交，而不是依赖提交后解释。
